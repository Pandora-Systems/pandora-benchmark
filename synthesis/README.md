# `synthesis/` — Document Map

The `synthesis/` folder defines the reasoning logic of Pandora.

It explains how structured analytical outputs become structured intelligence, how signal separation is preserved after measurement, how synthesis remains distinct from both layers and workflows, and how Pandora turns valid signals into higher-order judgments without collapsing into hidden evaluation.

If `framework/` defines structure and `workflows/` define execution, `synthesis/` defines how Pandora reasons once valid analytical outputs already exist.

This folder should be read as the reasoning backbone of Pandora. It contains the core synthesis doctrine first, followed by the modular synthesis block system that applies specific interpretive functions to structured outputs.

# Full Document Map

## Core Synthesis Documents

## 1. `synthesis-overview.md`

Defines the role of synthesis inside Pandora. This document explains why synthesis exists as a distinct reasoning domain, where it sits in the analysis lifecycle, how it differs from layers and workflows, and what higher-order analytical functions it is responsible for.

---

## 2. `synthesis-principles.md`

Defines the governing principles of Pandora’s reasoning domain. This document explains how synthesis must remain evidence-anchored, explicit, post-measurement, traceable, modular, and distinct from both evaluation and execution.

---

## 3. `truths-of-relativity.md`

Defines the principle that analytical outputs should be treated as frame-relative truths until synthesis proves otherwise. This document prevents premature contradiction assignment by requiring frame, metric, level, and context alignment before apparent opposition is treated as real contradiction.

---

## 4. `block-model.md`

Defines the conceptual unit of synthesis: the synthesis block. This document explains what a synthesis block is, how it differs from layers and workflows, what makes a block valid, and why Pandora uses modular reasoning units instead of one monolithic synthesis phase.

---

## 5. `block-interface.md`

Defines the formal contract every synthesis block must expose. This document explains the required fields, inputs, outputs, dependency logic, scope declarations, traceability requirements, prohibited actions, and null-state discipline that make synthesis blocks safe to reuse and compose.

---

## 6. `block-composition-rules.md`

Defines how synthesis blocks may be combined without breaking Pandora’s principles. This document explains dependency discipline, forward flow, scope preservation, ambiguity propagation, bias control, null-state preservation, and the rules that prevent synthesis chains from collapsing into hidden logic.

# `synthesis/blocks/` — Synthesis Block Definitions

The `blocks/` folder contains the concrete definitions of individual synthesis blocks.

These documents are not the core doctrine of synthesis itself. They are the operational block definitions that apply specific interpretive functions to structured upstream outputs.

Each block should comply with the principles, model, interface, and composition rules defined in the core synthesis documents.

Typical block documents in this folder include:

## `contradiction-detection.md`
Defines how Pandora identifies, classifies, and interprets contradiction across valid signals.

## `correlation-logic.md`
Defines how Pandora identifies reinforcing or convergent relationships across outputs.

## `severity-synthesis.md`
Defines how Pandora forms severity judgments from structured signal combinations.

## `final-assessment-logic.md`
Defines how Pandora produces integrated analytical judgments from prior synthesis results.

## `consensus-protocols.md`
Defines how multiple valid outputs are reconciled when more than one evaluator, pass, or synthesis path exists.

## `disagreement-analysis.md`
Defines how disagreement itself becomes a meaningful synthesis signal.

## `reporting-protocols.md`
Defines how synthesized intelligence is rendered into different reporting modes without changing its analytical substance.

## `unified-template.md`
Defines the standardized output pattern for synthesized conclusions.

Additional synthesis blocks may be added later, provided they remain consistent with the core synthesis doctrine.

# Folder Logic

The internal logic of `synthesis/` is therefore split into two parts:

## Core synthesis doctrine
The root files in `synthesis/` define:
- what synthesis is,
- what constrains it,
- how frame-relative truth is preserved,
- what a synthesis block is,
- what a block must declare,
- and how blocks may be composed.

## Block-level reasoning modules
The `synthesis/blocks/` folder defines:
- what each specific synthesis block does,
- what signals it consumes,
- what it emits,
- and how it contributes to final reasoning.

This split keeps the folder clean:
- the root defines the reasoning system,
- the `blocks/` folder defines the individual reasoning units.
